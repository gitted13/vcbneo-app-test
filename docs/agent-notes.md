## 2026-07-15 session (same agent, continued further) — delete-type feature + full DateRules rewrite

### Context
User pointed out a duplicate "Core Banking" type on the deployed server they didn't create. While that was being root-caused, user also asked (mid-turn, on top of a screenshot of the DateRules/"Phân loại trạng thái đối soát" page) for a real revamp of that page: no way to create a new status, editing is a tiny inline expand instead of a modal, need AND/OR condition groups, and translate the raw field/operator/value codes into something an end user can actually read instead of code-naming.

### Part A — Root cause: stray duplicate file type

`backend/app/db/seed.py`'s `seed_flex()` runs on every backend startup (documented as safe/idempotent) and matched existing rows by **`upload_name`** (the user-editable display name) instead of the stable `type_code` inside `fields_schema`. It also **unconditionally overwrote `fields_schema`** on every match. Two bugs from one root cause:
1. User renamed "Core Banking" → "Core Banking đi" via FileTypeSettings. Next restart, the seeder looked for a row named exactly "Core Banking", found none (it got renamed), and inserted a fresh duplicate with blank defaults — exactly the stray row in the screenshot.
2. For any type *not* renamed, the seeder was silently reverting column customizations to the hardcoded seed defaults on every single restart — a much bigger latent bug than the visible duplicate.

**Fixed**: match by `type_code` (`JSON_VALUE(fields_schema,'$.type_code')`) instead of `upload_name`; only `INSERT` when truly missing, **never** `UPDATE` an existing row — seed data is for a fresh DB, not a sync-on-every-boot. Reproduced the exact bug locally (renamed a type via the real API, restarted the backend, watched a duplicate appear under the old code — 8 types total, matching the screenshot), then confirmed the same rename+restart sequence produces **no** duplicate under the fix (still 7 types). Test rename was restored afterward.

### Part B — DateRules full rewrite

Before touching anything, read `frontend/src/data/reconcile.js` in full — this uncovered that the labels DateRules showed for each status row (`SWIFT_COLS_DI`, `CORE_COLS_DI`, `NAPAS_COLS_DI`, etc.) are the **same exported arrays** `CoreSummary`/`NapasCore`/`SwiftCore` import for their KPI-tile `filterFn` predicates against real `get_db_rows()` data — a completely unrelated, currently-working feature. Those arrays were never touched. The actual reason "create new" was impossible: DateRules' editable conditions (`reconcileStatusRules` in the DB) and their displayed labels/colors (`data/reconcile.js`) and their internal status codes (`engine_flex.py`'s `_STATUS_COLS`) were **three separate arrays kept in sync only by matching array index** — no rule had its own identity anywhere, so adding/removing a row would silently desync everything. This positional coupling was already fragile before this session: `SWIFT_DEN` has 6 status codes but 7 displayed rows, and `NAPAS_DEN` has 3 status codes but `NAPAS_COLS_DEN` (4 entries) was what DateRules actually rendered as its row count — the 4th row's condition editor was permanently dead (`rowConds[i] ?? []` → always empty, unreachable by the engine). Not introduced this session, just newly understood.

**New data model** — `reconcileStatusRules.rules_json[cond_key]` is now a list of self-contained rule objects instead of a bare list of chip-lists:
```json
{"id": "swift_di_0", "label": "Thành công – Core ngày T", "color": "#059669",
 "groups": [[{"f":"TT Swift","op":"=","v":"Thành công"}, ...], [...]]}
```
`groups` is OR'd together, chips within a group are AND'd — this is the AND/OR capability asked for, and a plain single-group rule with N chips is exactly what a migrated legacy rule looks like (backward compatible by construction).

**Backend** (`engine_flex.py`):
- `_classify()` no longer takes a `status_cols` positional-name array; it iterates rule objects directly and returns the matching rule's own `label` — the human-readable name **is** the status value now, everywhere downstream (`reconcileResults.status`, `flex-summary` breakdown, etc.), not a separate internal code. Deleted `_STATUS_COLS` entirely (confirmed via repo-wide grep no other file referenced it).
- `_eval_rule_groups()` — new, `any(all(chip matches) for group in groups)`. `_eval_chip()` (the per-chip evaluator, incl. the diacritics-normalization fix from earlier this session) is untouched.
- `migrate_status_rules(rules) -> (rules, changed)` — auto-upgrades any cond_key still in the old bare-list format, generating label/color from `_LEGACY_RULE_META` (a transcription of the old `data/reconcile.js` labels into Python, since the backend can't import a JS file — used *only* as migration seed text, not at runtime otherwise). Idempotent, preserves existing conditions exactly (wraps them as a single OR-group) — including the two pre-existing gaps mentioned above (an empty NAPAS-KTC condition list, and a self-contradictory `Core≠null AND Core=null` CORE_DEN rule that could never match). Both are now visible and editable through the new UI instead of being silent, undiscoverable dead logic. Called from both `run_flex_reconcile()` and `GET /status-rules` — whichever runs first migrates and persists the upgrade back to the DB, so it only happens once.
- `backend/app/db/seed.py`'s `_STATUS_RULES` rewritten directly in the new format (for fresh installs, no migration needed) — and the self-contradictory `CORE_DEN` rule fixed to its clearly-intended single condition while rewriting it (`[Core=null]` only).

**Frontend**:
- New file `frontend/src/data/dateRulesVocab.js` — human-label dictionaries for fields (`FIELD_LABELS`), known status values (`STATUS_VALUE_OPTIONS`), and `describeChip`/`describeGroup`/`describeRule` sentence builders. Deliberately a *new* file, not a change to `data/reconcile.js`, to keep this fully decoupled from the CoreSummary/NapasCore/SwiftCore KPI-tile system. Field kind (`presence`/`date`/`status`) mirrors `_eval_chip()`'s own branching exactly, so the editor only ever offers operations the engine actually understands.
- `frontend/src/pages/DateRules/index.jsx` — full rewrite. Rows render from `conds[condKey]` directly (each rule's own `label`/`color`/`groups`), not from the old `SECTIONS[...].tabs[...].cols` arrays. New `RuleModal` (proper `<Modal>`, not inline expand) with a group-of-groups editor: add/remove AND-conditions within a group, add/remove OR-groups, field dropdown drives kind-appropriate inputs (presence → Có/Không select; date → operator + other-date-field picker; status → operator + known-value picker), a 10-color preset swatch picker, and a delete button for the whole rule. "+ Tạo trạng thái mới" per tab. `DEFAULT_CONDS` (for "Khôi phục mặc định") rewritten to mirror the new `seed.py` exactly.

**Verified end-to-end against real local data**:
- `GET /status-rules` migration: confirmed correct shape, and confirmed it faithfully preserved every existing condition including the two pre-existing gaps described above (did not silently "fix" the user's live data).
- Re-ran `POST /reconcile/run-flex` for config_id=3 (same config as the diacritics-fix verification earlier) post-migration: **identical distribution** (6,939/2,415/18/5/5), now labeled with human text ("Thành công – Core ngày T") instead of the old code ("TC_KHOP") — confirms the full data-model rewrite didn't change classification behavior.

- Directly unit-tested `_eval_rule_groups()` OR-of-AND semantics with synthetic cases (group-1-only match, group-2-only match, neither matches, AND-within-group true/false) — all correct.
- `frontend/src/pages/Reconcile/index.jsx`: since `status` is now a free-text human label instead of a fixed enum, `statusMeta()` needed a way to color statuses it doesn't recognize from the old hardcoded `STATUS_META` dict. Added a live fetch of `GET /status-rules` on mount, flattened into a `label → color` map (`_dynamicStatusColors`), used as a fallback before the generic gray default — so new-format results get the *actual* color the rule author picked, not a guess. `STATUS_META` itself untouched, still the fallback path for old reconcileResults rows computed before this deploys.
